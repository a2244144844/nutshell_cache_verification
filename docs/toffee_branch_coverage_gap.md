# Toffee Branch Coverage Reporting Gap

Date: 2026-05-30 | Discovered by: Human-AI collaborative review

## Overview

During a human review of the coverage HTML reports generated by `scripts/collect_coverage.sh`, a discrepancy was identified: the LCOV HTML report at `build/reports/line_dat/picker_cache/index.html` shows **85.0% branch coverage (24,611 / 28,949)**, while the same toffee-test pipeline produces `build/reports/line_dat/code_coverage.json` with **95.3% branch coverage (471 / 494)**. This document records the root cause analysis, the exact data provenance of both numbers, the pipeline gap, and the recommended fix.

## Root Cause: Two Data Flows, One HTML

The `collect_coverage.sh` script triggers toffee-test's `convert_line_coverage()` function (`.venv/.../toffee_test/utils/__init__.py` line 31-39), which performs two separate operations on the same Verilator `.dat` coverage files:

### Flow A — LCOV HTML generation (→ 85% branch coverage)

```python
# toffee_test/utils/__init__.py, line 34
su, so, se = exe_cmd(["genhtml", "--branch-coverage", merged_info, "-o", output_dir])
```

1. Reads Verilator `.dat` files → raw coverage entries (C++ simulation level)
2. Calls `verilator_coverage` CLI → generates `merged.info` in LCOV format
3. Calls `genhtml` on `merged.info` → generates `line_dat/index.html`

**Result**: 28,949 branches counted at the **Verilator C++ simulation kernel level**, where a single RTL `wire` declaration can generate 128 C++ conditional branches.

### Flow B — RTL-level JSON extraction (→ 95.3% branch coverage)

```python
# toffee_test/utils/verilator_coverage/processor.py, line 40
verilator_coverage_miss(filtered_coverage, os.path.join(output_dir, "code_coverage.json"))
```

1. Reads the same Verilator `.dat` files
2. Parses each entry using `VerilatorCoverage` class, which extracts the `\x01page\x02v_branch/ModuleName` metadata
3. Filters for entries where `meta.type == "branch"` — i.e., Verilator's own classification of RTL-level branch constructs
4. Groups by RTL module name and counts per-module hit/miss
5. Writes `code_coverage.json`

**Result**: 494 branches counted at the **RTL module level**, correctly reflecting the design's logical branch structure.

### The Gap

`code_coverage.json` (Flow B) has the correct RTL-level branch data at 95.3%, but **no HTML visualization is generated from it**. The only HTML that shows branch coverage is Flow A's LCOV output, which uses C++-level data at 85.0%.

Both flows run inside the same `convert_line_coverage()` function (line 33 calls `convert_verilator_coverage()` which runs flow B internally, then line 34 runs flow A). The correct RTL branch data is computed and saved, then immediately overshadowed by a less meaningful C++-level HTML.

## Data Provenance: Verifying the Two Numbers

### Evidence 1: RTL line → C++ branch explosion

A scan of `merged.info` reveals that 1,143 of 1,364 RTL lines carry C++ branch annotations. A single non-branch RTL line demonstrates the scale of the mismatch:

```
Cache.v line 8:  input [63:0] io_in_bits_wdata,  ← port declaration, zero logic branches
                 ↓ Verilator compilation
                 128 C++ branches attributed (64-bit bus → many C++ conditionals)
```

The top C++-branch-dense lines:

| RTL Line | C++ Branches | RTL Content (typical) |
|---|---|---|
| 532 | 130 | `dataRead = useForwardData ? ... : ...` |
| 543 | 130 | `wordMask = cmd[0] ? ... : 64'h0` |
| 8 | 128 | `input [63:0] io_in_bits_wdata,` |
| 16–107 | 128 each | Port declarations, wire assigns |

**Total**: 28,949 C++ branches from 1,364 RTL lines. Ratio: ~21 C++ branches / RTL line.

### Evidence 2: VerilatorCoverage type filter

In `toffee_test/utils/verilator_coverage/models.py` (line 43-44):

```python
type_part, module_part = v.split("/")
self.type = type_part.lstrip("v_")   # → "branch", "line", "toggle", "expr"
self.module_name = module_part       # → "CacheStage3", "Cache", etc.
```

In `processor.py` (line 67-83), only entries with `meta.type == "branch"` increment the branch counter. Verilator's own `.dat` metadata distinguishes between C++ instrumentation points (`page=v_branch/ModuleName` → RTL branch) and general instrumentation noise. This filtering extracts exactly 494 RTL-level branches.

### Evidence 3: Module-level breakdown in code_coverage.json

| Module | Branch Total | Branch Hit | Branch Cov% | Notes |
|---|---|---|---|---|
| Cache | 66 | 65 | 98.5% | 1 uncovered (line 2674: response gating mux) |
| CacheStage1 | 0 | 0 | — | Combinational, no branch constructs |
| CacheStage2 | 58 | 48 | 82.8% | 10 uncovered (forward-meta paths, assertion check) |
| CacheStage3 | 210 | 198 | 94.3% | 12 uncovered (probe state, assertion checks, beat counters) |
| Arbiter ×5 | 26 | 26 | 100% | |
| SRAMTemplate ×4 | 134 | 134 | 100% | |
| **TOTAL** | **494** | **471** | **95.3%** | |

This module-level grouping is only possible because Verilator's `.dat` format tags each coverage entry with the originating Verilog module — a capability that LCOV's flat file-per-source format (`merged.info`) does not support, causing the C++-level data to collapse into a single `Cache.v` bucket.

## Recommended Fix

### Short-term (submission-ready workaround)

A reusable Python script generates the RTL branch coverage HTML from `code_coverage.json`:

```sh
# Run standalone
python scripts/generate_rtl_coverage_html.py

# Or specify paths
python scripts/generate_rtl_coverage_html.py -i build/reports/line_dat/code_coverage.json -o build/reports/rtl_coverage.html
```

The script is also integrated into `scripts/collect_coverage.sh` so it runs automatically after every coverage collection.

The generated `build/reports/rtl_coverage.html` visualizes per-module Branch / Line / Toggle / Expr coverage with expandable uncovered-line details. It should be referenced in the submission README and coverage documents as the authoritative branch coverage visual.

### Long-term (toffee-test improvement proposal)

Modify `toffee_test/utils/verilator_coverage/__init__.py` to add an RTL-level HTML generation step after `verilator_coverage_miss()`:

```python
def convert_verilator_coverage(line_coverage_list, output_dir):
    # ... existing code ...
    verilator_coverage_miss(filtered_coverage, os.path.join(output_dir, "code_coverage.json"))
    merged_info = os.path.join(output_dir, "merged.info")
    verilator_coverage_to_lcov(filtered_coverage, merged_info)
    
    # NEW: generate RTL-level branch HTML from code_coverage.json
    generate_rtl_coverage_html(
        os.path.join(output_dir, "code_coverage.json"),
        os.path.join(output_dir, "rtl_branch_coverage.html")
    )
    return merged_info, ignore_info
```

Or alternatively, generate a module-filtered LCOV file (using only entries where `type == "branch"`) and run `genhtml` on that, producing a `line_dat_rtl/` directory alongside `line_dat/`.

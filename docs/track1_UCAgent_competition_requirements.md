# 第八届 CCF 开源创新大赛 Track1 UCAgent 赛事信息整理

## 1. 赛题定位

本赛题以“从架构认知到工业级验证实战”为主线，目标不是简单使用 AI 生成代码，而是让参赛者完整经历从 RTL 逻辑理解、验证环境搭建、随机激励设计、覆盖率分析到缺陷追踪的硬件验证工程流程。

参赛过程强调 **UCAgent 辅助 + 人工深度定制**。UCAgent 可以用于快速生成验证原型、辅助理解设计、提升编码和定位效率，但最终作品必须体现开发者在验证策略、约束设计、环境重构、Bug 场景设计和报告分析中的主导作用。

## 2. 技术学习目标

### 2.1 验证原理深度剖析

参赛者需要系统掌握以下验证核心概念：

- 验证层次化模型
- 受约束随机激励，CRV，Constrained Random Verification
- 功能覆盖率，Functional Coverage
- 测试计划、覆盖率闭环和 Bug 追踪

### 2.2 跨语言仿真赋能：Picker

参赛者需要学习并使用 Picker 工具，将复杂 RTL 设计转换为高性能 Python 驱动模块，从而打通 HDL 与 Python 等高级通用语言之间的边界。

Picker 在本赛题中的核心作用：

- 将硬件逻辑转换为可由 Python 驱动的仿真模块
- 降低 RTL 交互门槛
- 支撑更灵活的激励生成、检查和自动化验证流程

### 2.3 结构化验证环境构建：Toffee

参赛者需要基于 UCAgent 关联的 Toffee 框架构建现代化、可复用、易维护的验证环境。

Toffee 在本赛题中的核心作用：

- 组织验证组件
- 管理激励发生器、监视器、计分板等模块
- 支撑类似软件测试的敏捷验证流程
- 提升验证工程的结构化程度和可维护性

## 3. 核心任务

### 3.1 命题

基于 UCAgent 完成对 **果壳 NutShell Cache** 的自动化验证。

### 3.2 需要提交的成果

参赛者需要提交以下内容：

- **代码仓库**：包含针对果壳 Cache 编写的完整验证组件、激励发生器 Generator、计分板 Scoreboard 等内容。
- **验证报告**：涵盖测试计划、覆盖率分析、Bug 追踪记录、AI 协同过程和人工修正过程。

## 4. 开发者任务重心

参赛者可以利用 UCAgent 快速构建验证原型，但必须通过人工介入进行深度定制。重点包括但不限于以下方向。

### 4.1 约束细化

人工编写复杂的受约束随机激励，覆盖 UCAgent 难以自动触达的 Corner Case。

建议重点关注：

- Cache 读写路径
- Cache miss / hit 场景
- 替换算法边界情况
- 脏数据写回场景
- 地址冲突和随机访问压力
- 一致性或状态转换相关 Corner Case

### 4.2 架构重构

将 UCAgent 生成的代码片段整合进 Toffee 的结构化框架中，并进行人工 Code Review 与重构。

应体现：

- 清晰的验证组件分层
- 可维护的目录结构
- 可复用的 Generator、Driver、Monitor、Scoreboard
- 明确的测试入口和回归方式
- 对 AI 生成代码的人工修正记录

### 4.3 故障注入

人工设计特定 Bug 场景，用于校验验证环境的检出能力。

建议记录：

- 注入的 Bug 类型
- 触发条件
- 验证环境如何观测到异常
- Scoreboard 或 assertion 如何定位问题
- 修复前后的仿真与覆盖率对比

## 8. 奖项等级判定

### 8.1 一等奖：AI 创新奖，85-100 分

定位：资深验证专家视角。

实操表现：

- 验证环境具有较高人工逻辑密度。
- 代码不是简单 AI 堆砌，而是经过开发者深度校验和重构后的工业级代码。
- 成功验证并证明果壳 Cache 在特定压力场景下的逻辑正确性。

报告深度：

- 包含详尽的“AI 缺陷与人工修正对比表”。
- 清晰体现开发者在验证策略制定、关键 Bug 拦截中的主导作用。

### 8.2 二等奖 / 参与奖，60-84 分

定位：合格工程师视角。

实操表现：

- 完成核心功能验证。
- 虽然代码细节可能保留 AI 生成痕迹，但经过基础人工校验。
- 仿真结果真实有效。

报告深度：

- 描述基本使用流程。
- 能说明人工干预的必要性。
- 附件和复现实验材料齐全。

## 9. 基础参评要求，一票否决项

### 9.1 开源合规性

参赛代码仓库必须严格符合 Apache 2.0 开源协议。

仓库根目录需要包含标准 `LICENSE` 文件。

### 9.2 工程规范性

需遵循约定俗成的文件目录结构，例如：

- `src/`
- `tests/`
- `docs/`

禁止提交无意义的冗余中间件、临时文件或不可解释的生成物。

### 9.3 严禁纯 AI 刷票

评审组将对提交代码进行深度审计。

若作品存在以下问题，将视为无效作品，不予参评：

- 代码仅为 UCAgent 原始输出。
- 缺乏人工逻辑注入。
- 无法处理复杂验证场景。
- 存在明显 AI 逻辑谬误且未修正。
- 缺少人工 Code Review、调试和验证闭环记录。

## 12. 建议执行路线

### 阶段一：理解设计与建立最小闭环

- 阅读 NutShell Cache 相关 RTL 和接口文档。
- 使用 UCAgent 辅助总结 Cache 结构、接口信号、关键状态机和数据路径。
- 使用 Picker / Toffee 建立最小可运行验证闭环。
- 完成 smoke test，证明环境可以驱动 DUT 并收集结果。

### 阶段二：构建结构化验证环境

- 拆分 Generator、Driver、Monitor、Scoreboard。
- 设计基础读写、命中、缺失、替换、写回等测试。
- 将 UCAgent 生成内容纳入 Toffee 风格工程结构。
- 对 AI 生成代码进行人工审查和重构。

### 阶段三：强化随机与 Corner Case

- 设计受约束随机激励。
- 覆盖地址冲突、连续 miss、替换边界、脏块写回、随机读写压力等场景。
- 引入 coverage model，跟踪功能覆盖率。
- 针对覆盖率缺口补充 directed test。

### 阶段四：故障注入与检出能力证明

- 人工设计若干 Bug 场景。
- 记录 Bug 触发方式、检测路径和定位过程。
- 证明 Scoreboard、Monitor 或 assertion 能发现异常。
- 将 Bug 追踪记录写入报告。

### 阶段五：整理报告与复现材料

- 输出测试计划。
- 输出覆盖率报告。
- 输出 Bug 追踪记录。
- 输出 AI 协同分析，说明 UCAgent 生成了什么、人工改了什么、为什么要改。
- 整理一键运行脚本和复现说明。

## 15. 对当前开发工作的启示

本赛题可以使用 UCAgent 辅助完成验证任务，但最终得分重点不在“是否调用了 AI”，而在于：

- 是否能搭建可运行、可复现、可维护的验证环境。
- 是否能体现人工对 Cache 架构和验证策略的理解。
- 是否能用 CRV、coverage、scoreboard、bug injection 等方法证明验证质量。
- 是否能清楚记录 AI 协同过程，并指出 AI 输出中的不足与人工修正。

因此，实际开发时应把 UCAgent 作为验证工程助手，而不是最终答案生成器。高分作品需要展示“AI 加速原型，人工完成工程化与深度验证”的完整过程。
